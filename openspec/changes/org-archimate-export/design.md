# Design: org-archimate-export

## Context

Extends the partially-implemented softwarecatalog ArchiMate organization export to add:
- Deelnames data layer (RBAC-bypassed query for participations in other orgs' software)
- Typed organization folders in the AMEFF `<organizations>` section
- Liberal boolean query-parameter parsing (`1`, `yes`, `true`, and case variants)
- File name sanitization for org names with special characters
- Frontend data-layer toggle checkboxes with loading state

## Goals / Non-Goals

**Goals:**
- Complete all requirements in `specs/org-archimate-export/spec.md`
- Produce a valid AMEFF XML file that imports into Archi without errors
- Expose deelnames data via the existing GET endpoint with a new `?deelnames=true` parameter
- Organize SWC objects into typed folders matching the GEMMA Online convention
- Make the frontend reflect the three data layers with checkboxes

**Non-Goals:**
- Modifying the base (non-organization) ArchiMate export
- Streaming large responses (in-memory assembly is adequate for current volumes)
- Round-trip XML import

## Architecture Overview

```
Frontend (GET + query params)
  → SettingsController::exportOrgArchiMate(organizationUuid)
      → parse modules/deelnames/gebruik params (liberal truthy)
      → sanitize org name for filename
      → ArchiMateService::exportOrgArchiMate(orgUuid, options)
            → query org from Voorzieningen register
            → if options[modules]: query org's gebruik + modules
            → if options[deelnames]: searchObjects(_rbac=false, deelnemers=orgUuid)
            → ArchiMateExportService::exportOrganizationArchiMateXml(
                  objectService, registerId, schemaIdMap,
                  orgName, orgUuid,
                  gebruikData, modulesData, deelnamesData, options
              )
                → base GEMMA objects
                → generateApplicationElements() per type (gebruikt, aangeboden, deelnames)
                → generateSpecializationRelationships() per element
                → ensureBronPropertyDefinition()
                → copyAndEnrichViews()
                → buildSwcOrganizationFolders(typed buckets)
                → assembleOrganizationXml()
            → returns {xml, filename}
      → returns XML as file download (Content-Disposition with sanitized filename)
```

## API Design

### `GET /api/archimate/export/organization/{organizationUuid}`

**Path parameter:**
| Parameter | Type | Description |
|-----------|------|-------------|
| `organizationUuid` | UUID string | UUID of the organisation to export |

**Query parameters (all optional):**
| Parameter | Type | Default | Description |
|-----------|------|---------|-------------|
| `modules` | bool | `true` | Include org's own modules as ApplicationComponent elements |
| `deelnames` | bool | `false` | Include deelname gebruik (RBAC-off query for participations) |
| `gebruik` | bool | `false` | Include org's aangeboden gebruik data |

Truthy detection: any of `1`, `yes`, `true`, `on` (case-insensitive). Everything else is falsy.

**Success response:**
```
HTTP/1.1 200 OK
Content-Type: application/xml
Content-Disposition: attachment; filename="20-05-2026_Softwarecatalogus_AMEFF_export_Zeist.xml"

<?xml version="1.0" encoding="UTF-8"?>
<model xmlns="http://www.opengroup.org/xsd/archimate/3.0/" ...>
  ...
</model>
```

**Error responses:**
```json
// 404 — org not found
{ "success": false, "message": "Organization not found", "error": "NOT_FOUND" }

// 401 — unauthenticated
{ "success": false, "message": "Not authorized", "error": "UNAUTHORIZED" }

// 403 — insufficient permissions
{ "success": false, "message": "Not authorized", "error": "FORBIDDEN" }
```

## Key Design Decisions

### Decision 1: Liberal truthy parsing for boolean query parameters

**Choice:** Accept `1`, `yes`, `true`, `on` (any case) as truthy; everything else is falsy.

**Rationale:** The context-brief explicitly requires `1`, `yes`, `true`, `TRUE` to be accepted. URLs generated by different tools (curl, Postman, browser) use different conventions. A single `strtolower($val)` check against `['1', 'yes', 'true', 'on']` is adequate.

```php
private function parseBool(string $value, bool $default = false): bool {
    if ($value === '') return $default;
    return in_array(strtolower($value), ['1', 'yes', 'true', 'on'], true);
}
```

### Decision 2: File name sanitization (Content-Disposition only)

**Choice:** Replace characters outside `[A-Za-z0-9._-]` in the filename with `_`; preserve the original organization name in the XML model `<name>`.

**Rationale:** Content-Disposition filenames must be ASCII-safe. Special characters like `'` and `-` in `Gemeente 's-Hertogenbosch` produce browser-specific filename handling. The model name inside the XML can and should preserve the original for display in Archi.

```php
$safeOrgName = preg_replace('/[^A-Za-z0-9._-]/', '_', $orgName);
$filename = date('d-m-Y') . '_Softwarecatalogus_AMEFF_export_' . $safeOrgName . '.xml';
```

### Decision 3: Typed organization folders

**Choice:** Produce separate subfolders under the org label, one per data type. Empty folders are omitted.

**Rationale:** Architects in Archi need to distinguish "software Zeist uses" from "software Zeist provides" from "participations". A single `Applicaties (Softwarecatalogus)` folder mixes all three and requires property inspection to tell them apart. Typed folders make the distinction immediately visible in the Archi model tree.

```xml
<item>
  <label>Zeist</label>
  <!-- Present only when data exists for that type -->
  <item><label>Gebruikt (Softwarecatalogus)</label>...</item>
  <item><label>Aangeboden (Softwarecatalogus)</label>...</item>
  <item><label>Deelnames (Softwarecatalogus)</label>...</item>
  <item><label>Relaties (Softwarecatalogus)</label>...</item>
  <item><label>Views (Softwarecatalogus)</label>...</item>
</item>
```

### Decision 4: Deelname query uses RBAC-off ObjectService search

**Choice:** Query gebruik objects with `_rbac: false` and `_multitenancy: false` to find records where `deelnemers` contains the current org UUID.

**Rationale:** Deelname gebruik objects are owned by other organisations. Standard RBAC would filter them out. The same bypass pattern is already used in `ViewService::getGebruikData()` step 2. The query is explicitly opt-in via `?deelnames=true`; no RBAC data is exposed by default.

```php
$deelnameQuery = [
    '@self' => ['register' => $orgRegisterId, 'schema' => $gebruikSchemaId],
    'deelnemers' => $organizationUuid,
    '_limit' => 10000
];
$deelnamesData = $objectService->searchObjects(
    query: $deelnameQuery, _rbac: false, _multitenancy: false
);
```

### Decision 5: Deterministic element identifiers

**Choice:** Identifiers are derived from object UUIDs (`id-swc-app-{uuid}`, `id-swc-rel-{appUuid}-{refcompId}`, `id-swc-view-{originalViewId}`). No random generation.

**Rationale:** Re-importing the same export twice into Archi must not create duplicate elements. Deterministic identifiers mean the second import updates existing objects rather than duplicating them.

### Decision 6: XML character escaping

**Choice:** Use PHP's `htmlspecialchars()` on all user-supplied string values inserted into XML. Do not trust the existing DOMDocument node; rebuild text nodes through the DOM API.

**Rationale:** Module names may contain `&`, `<`, `>`, or `"`. Inserting them as raw text into an XML string produces malformed output. The DOM API handles escaping automatically when using `createElement`/`createTextNode`.

### Decision 7: Frontend uses NcCheckboxRadioSwitch with try/finally for loading state

**Choice:** Three `NcCheckboxRadioSwitch` components bound to `includeModules`, `includeDeelnames`, `includeGebruik`. Export button has a `loading` data property toggled via `try/finally`.

**Rationale:** Follows ADR-004 (no hardcoded strings, use NC components, `try/finally` for loading state). `NcCheckboxRadioSwitch` is the Nextcloud-idiomatic checkbox. The loading state prevents double-clicks during the download.

## Reuse Analysis

Per ADR-001 requirement (Deduplication Check), existing services leveraged:

| Service / Pattern | How reused |
|---|---|
| `ObjectService::searchObjects(_rbac: false)` | Deelname query; same pattern as `ViewService::getGebruikData()` step 2 |
| `ArchiMateExportService::buildModuleLookupMaps()` | Process deelnames data through same lookup-map pipeline as owned modules |
| `ArchiMateExportService::generateApplicationElements()` | Called per data type (gebruikt, deelnames) with a type tag |
| `ArchiMateExportService::generateSpecializationRelationships()` | Called per element set |
| `ArchiMateExportService::ensureBronPropertyDefinition()` | Shared across all generated SWC objects |
| `ArchiMateExportService::copyAndEnrichViews()` | No change needed — already handles multiple element sets |

No new services or mappers are required. The platform's `ObjectService` handles cross-tenant queries via the RBAC bypass.

## Seed Data

This change does not introduce or modify OpenRegister schemas. The export reads from existing schemas (Voorzieningen register: organisations, modules, gebruik; AMEF register: GEMMA objects). No seed data section required per ADR-001 exception: "Changes that only modify frontend components or non-schema backend logic do not require seed data."

## Security Considerations

- **Auth**: The endpoint requires Nextcloud authentication. No `#[PublicPage]` annotation. `#[NoAdminRequired]` is appropriate — any authenticated user can export (same as existing behavior).
- **Per-object IDOR**: The export reads all GEMMA data (public within the instance) filtered by the provided organization UUID. No cross-organization application data is exposed since modules and gebruik are queried per org. Deelname query is org-scoped (`deelnemers => $uuid`).
- **RBAC bypass scope**: The deelname bypass is limited to the deelname query only and is opt-in (`?deelnames=true`). The bypass is documented as intentional in `ViewService` context.
- **UUID validation**: The `organizationUuid` path parameter must be validated as a UUID before any database query to prevent injection.
- **No PII in responses**: Error messages use static strings per ADR-005. No stack traces or internal paths in 4xx/5xx responses.
- **XML escaping**: All user-supplied strings inserted into XML nodes must be escaped via DOM API (not raw string concatenation).

## NL Design System

Not applicable to the backend export logic. The frontend addition (three checkboxes + loading button) uses Nextcloud CSS variables and NL Design System tokens via `@conduction/nextcloud-vue`. WCAG AA: checkboxes are labelled via `NcCheckboxRadioSwitch` built-in label prop; the export button communicates loading state visually and via aria-busy.

## Trade-offs

### Considered: Separate endpoint for deelnames export
**Rejected**: A single endpoint with query parameters is simpler for frontend integration, avoids endpoint proliferation, and is idiomatic REST (filter the same resource, don't create a new one).

### Considered: All data layers enabled by default
**Rejected**: Deelnames and gebruik queries can be slow for orgs with many participations. Keeping them opt-in preserves backward compatibility and avoids surprising users with large exports.

### Considered: Streaming XML for large exports
**Deferred**: The current approach assembles the full XML in memory before sending. For organizations with 200+ modules across 50 views, this may exceed typical PHP memory in degenerate cases. The context-brief requires streaming only for files over 10 MB — log a warning if the assembled XML exceeds this threshold, but do not implement streaming in this change.
